Ejercicio 6 — API de Contactos Avanzada

Evolución del Ejercicio 5 · Nuevas funcionalidades y pruebas unitarias más completas

🔷
Prerrequisito: Ejercicio 5 completado

Este ejercicio parte del resultado del Ejercicio 5. Si aún no lo has hecho, complétalo primero. Aquí analizaremos sus limitaciones y agregaremos cinco mejoras concretas a la API, que luego deberás cubrir con pruebas unitarias.

Punto de partida — Ejercicio 5

La API del Ejercicio 5 cubre lo mínimo funcional: un CRUD básico sobre un array en memoria. Funciona, pero tiene cinco huecos importantes que encontrarás en cualquier API real.

Validación superficial

Solo se verifica que el email contenga @. Un string como "a@b" pasa la validación aunque no sea un email real.

Sin deduplicación

Puedes crear dos contactos con el mismo email. Esto rompería la búsqueda por email y generaría datos duplicados.

Sin búsqueda

No hay forma de filtrar contactos por nombre o email sin obtener toda la lista y filtrar en el cliente.

Sin acciones específicas

No existe forma de marcar un contacto como favorito sin sobreescribir todo el objeto con PUT.

Errores inconsistentes

Cada ruta genera mensajes de error con su propia estructura. No hay un formato uniforme ni un middleware centralizado.

Tests frágiles

Los tests dependen del ID 1 del array inicial. Si los datos de seed cambian, varios tests rompen en cascada.

¿Por qué extender la API?

No se trata solo de agregar funcionalidades — se trata de aprender cómo crecer código existente de forma segura. Cada mejora introduce un escenario nuevo que los tests deben cubrir.

Principio clave Cada nueva lógica de negocio necesita su propia prueba

Cuando un sistema crece sin nuevas pruebas, la cobertura se diluye: el porcentaje de líneas de código sin test aumenta con cada commit. Esto acumula "deuda de testing" que luego es muy costosa de pagar.

Las pruebas también actúan como documentación viva: cuando alguien nuevo lee it('rechaza emails duplicados con 409'), entiende inmediatamente que esa es una invariante del sistema, sin necesidad de leer el código de producción.

Las cinco mejoras — paso a paso

Primero lee y entiende cada mejora. Al final del documento encontrarás el código completo de la API extendida como referencia, y el ejercicio donde tú escribes todos los tests nuevos.

Mejora 1 Validación de email con expresión regular

Reemplazamos email.includes('@') por una regex que verifica el formato mínimo real: caracteres antes del @, un dominio y una extensión. Esto rechaza strings como "@", "a@b" o "test@".

🔷 Cambio en src/app.js — función validateContact
// ANTES (Ejercicio 5)
if (!email || !email.includes('@')) {
  return { valid: false, error: 'Email inválido.' };
}

// DESPUÉS (Ejercicio 6) — regex de formato básico
const EMAIL_REGEX = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;

function validateContact({ name, email }) {
  if (!name || name.trim() === '') {
    return { valid: false, error: 'El campo name es requerido.' };
  }
  if (!email || !EMAIL_REGEX.test(email)) {
    return { valid: false, error: 'El formato del email es inválido.' };
  }
  return { valid: true };
}
💡
Por qué esto importa para los tests

Ahora hay más casos de borde que probar: "@", "sin-arroba", "a@b" (sin dominio), "usuario@dominio." (punto al final). Cada caso es un test separado porque representa un camino diferente en la lógica de validación.

Mejora 2 Detección de email duplicado — 409 Conflict

Antes de crear un contacto, verificamos si ya existe uno con el mismo email (insensible a mayúsculas). Si hay duplicado, respondemos con 409 Conflict — el código HTTP correcto para este caso, distinto del 400 Bad Request que indica datos malformados.

🔷 Cambio en POST /api/contacts
app.post('/api/contacts', (req, res) => {
  const { name, email, phone } = req.body;

  const validation = validateContact({ name, email });
  if (!validation.valid) {
    return res.status(400).json({ error: validation.error });
  }

  // NUEVO: verificar duplicado (case-insensitive)
  const duplicate = contacts.find(
    c => c.email.toLowerCase() === email.toLowerCase()
  );
  if (duplicate) {
    return res.status(409).json({ error: 'Ya existe un contacto con ese email.' });
  }

  const newContact = {
    id: nextId++,
    name: name.trim(),
    email: email.toLowerCase(),
    phone: phone?.trim() || null,
    favorite: false,
    createdAt: new Date().toISOString(),
  };
  contacts.push(newContact);
  res.status(201).json(newContact);
});
💡
409 vs 400 — ¿por qué el código HTTP importa en los tests?

Testear el código de status exacto (expect(res.status).toBe(409)) documenta un contrato del API. Si alguien cambia el 409 por un 400, el test falla y el equipo sabe que rompió la semántica HTTP — algo que un test de "solo verifica que falle" no detectaría.

Mejora 3 Búsqueda y filtrado con query params

El endpoint GET /api/contacts ahora acepta dos query params opcionales: ?search=texto filtra por nombre o email (case-insensitive), y ?favorite=true filtra solo los marcados como favorito. Ambos se pueden combinar.

🔷 Cambio en GET /api/contacts
app.get('/api/contacts', (req, res) => {
  const { search, favorite } = req.query;

  let result = [...contacts];

  // Filtro por texto (nombre o email)
  if (search && search.trim() !== '') {
    const q = search.trim().toLowerCase();
    result = result.filter(
      c => c.name.toLowerCase().includes(q) ||
           c.email.toLowerCase().includes(q)
    );
  }

  // Filtro por favoritos
  if (favorite === 'true') {
    result = result.filter(c => c.favorite === true);
  }

  res.json(result);
});
💡
Cuántos tests genera este cambio

Al menos cinco: búsqueda con resultado, búsqueda sin resultado, búsqueda insensible a mayúsculas, filtro de favoritos, combinación de ambos filtros. Cada combinación de parámetros es un caso de uso distinto que puede romperse por separado.

Mejora 4 Endpoint de favoritos — PATCH semántico

En lugar de hacer un PUT con todo el objeto para cambiar solo el campo favorite, agregamos un endpoint dedicado PATCH /api/contacts/:id/favorite que lo alterna (toggle). Esto sigue el principio de verbos HTTP correctos y hace los tests más expresivos.

🔷 Nuevo endpoint PATCH /api/contacts/:id/favorite
// Toggle de favorito: si era false → true, si era true → false
app.patch('/api/contacts/:id/favorite', (req, res) => {
  const contact = contacts.find(c => c.id === Number(req.params.id));
  if (!contact) {
    return res.status(404).json({ error: 'Contacto no encontrado.' });
  }
  contact.favorite = !contact.favorite;
  res.json(contact);
});
💡
Tests de comportamiento toggle

Este endpoint requiere testear el ciclo completo: llamarlo una vez pone favorite: true, llamarlo de nuevo lo regresa a false. Un solo test no cubre ambas transiciones de estado.

Mejora 5 Middleware de error centralizado y formato uniforme

Agregamos un middleware de error al final de la app que captura errores no manejados y los devuelve en un formato JSON consistente. Además, todos los errores ahora incluyen un campo status en el body para facilitar el manejo en el cliente.

🔷 Middleware de error — al final de app.js
// Ruta no encontrada (404 genérico)
app.use((req, res) => {
  res.status(404).json({ status: 404, error: 'Ruta no encontrada.' });
});

// Manejador de errores inesperados
app.use((err, req, res, next) => {
  console.error(err.stack);
  res.status(500).json({ status: 500, error: 'Error interno del servidor.' });
});

module.exports = app;
💡
Testear rutas inexistentes es valioso

Un test que hace GET /api/ruta-que-no-existe y verifica status 404 con un cuerpo JSON específico documenta que la API nunca devuelve HTML de error por defecto — un bug frecuente en Express cuando no hay middleware de error configurado.

Node.js API extendida completa — src/app.js

Este es el código de producción de la API mejorada. Léelo con atención antes de escribir los tests — cada decisión tiene implicaciones en cómo se debe testear.

📄 src/app.js — versión completa (Ejercicio 6)
const express = require('express');
const app = express();
app.use(express.json());

// ── Validación de email con regex básica
const EMAIL_REGEX = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;

function validateContact({ name, email }) {
  if (!name || name.trim() === '') {
    return { valid: false, error: 'El campo name es requerido.' };
  }
  if (!email || !EMAIL_REGEX.test(email)) {
    return { valid: false, error: 'El formato del email es inválido.' };
  }
  return { valid: true };
}

// ── Datos iniciales en memoria
let contacts = [
  { id: 1, name: 'Ana García',   email: 'ana@example.com',  phone: '555-0001', favorite: false, createdAt: '2024-01-10T08:00:00.000Z' },
  { id: 2, name: 'Luis Pérez',   email: 'luis@example.com', phone: '555-0002', favorite: true,  createdAt: '2024-01-11T09:00:00.000Z' },
  { id: 3, name: 'Eva Martínez', email: 'eva@example.com',  phone: null,       favorite: false, createdAt: '2024-01-12T10:00:00.000Z' },
];
let nextId = 4;

// Permite a los tests reiniciar el estado
function resetContacts() {
  contacts = [
    { id: 1, name: 'Ana García',   email: 'ana@example.com',  phone: '555-0001', favorite: false, createdAt: '2024-01-10T08:00:00.000Z' },
    { id: 2, name: 'Luis Pérez',   email: 'luis@example.com', phone: '555-0002', favorite: true,  createdAt: '2024-01-11T09:00:00.000Z' },
    { id: 3, name: 'Eva Martínez', email: 'eva@example.com',  phone: null,       favorite: false, createdAt: '2024-01-12T10:00:00.000Z' },
  ];
  nextId = 4;
}

// ── GET /api/contacts  (con búsqueda y filtro de favoritos)
app.get('/api/contacts', (req, res) => {
  const { search, favorite } = req.query;
  let result = [...contacts];

  if (search && search.trim() !== '') {
    const q = search.trim().toLowerCase();
    result = result.filter(
      c => c.name.toLowerCase().includes(q) || c.email.toLowerCase().includes(q)
    );
  }

  if (favorite === 'true') {
    result = result.filter(c => c.favorite === true);
  }

  res.json(result);
});

// ── GET /api/contacts/:id
app.get('/api/contacts/:id', (req, res) => {
  const contact = contacts.find(c => c.id === Number(req.params.id));
  if (!contact) return res.status(404).json({ status: 404, error: 'Contacto no encontrado.' });
  res.json(contact);
});

// ── POST /api/contacts
app.post('/api/contacts', (req, res) => {
  const { name, email, phone } = req.body;

  const validation = validateContact({ name, email });
  if (!validation.valid) {
    return res.status(400).json({ status: 400, error: validation.error });
  }

  const duplicate = contacts.find(c => c.email.toLowerCase() === email.toLowerCase());
  if (duplicate) {
    return res.status(409).json({ status: 409, error: 'Ya existe un contacto con ese email.' });
  }

  const newContact = {
    id: nextId++,
    name: name.trim(),
    email: email.toLowerCase(),
    phone: phone?.trim() || null,
    favorite: false,
    createdAt: new Date().toISOString(),
  };
  contacts.push(newContact);
  res.status(201).json(newContact);
});

// ── PUT /api/contacts/:id (actualización parcial)
app.put('/api/contacts/:id', (req, res) => {
  const contact = contacts.find(c => c.id === Number(req.params.id));
  if (!contact) return res.status(404).json({ status: 404, error: 'Contacto no encontrado.' });

  const { name, email, phone } = req.body;

  if (email !== undefined) {
    if (!EMAIL_REGEX.test(email)) {
      return res.status(400).json({ status: 400, error: 'El formato del email es inválido.' });
    }
    const duplicate = contacts.find(
      c => c.id !== contact.id && c.email.toLowerCase() === email.toLowerCase()
    );
    if (duplicate) {
      return res.status(409).json({ status: 409, error: 'Ya existe un contacto con ese email.' });
    }
    contact.email = email.toLowerCase();
  }

  if (name !== undefined) {
    if (name.trim() === '') {
      return res.status(400).json({ status: 400, error: 'El campo name no puede estar vacío.' });
    }
    contact.name = name.trim();
  }

  if (phone !== undefined) contact.phone = phone?.trim() || null;

  res.json(contact);
});

// ── PATCH /api/contacts/:id/favorite (toggle)
app.patch('/api/contacts/:id/favorite', (req, res) => {
  const contact = contacts.find(c => c.id === Number(req.params.id));
  if (!contact) return res.status(404).json({ status: 404, error: 'Contacto no encontrado.' });
  contact.favorite = !contact.favorite;
  res.json(contact);
});

// ── DELETE /api/contacts/:id
app.delete('/api/contacts/:id', (req, res) => {
  const index = contacts.findIndex(c => c.id === Number(req.params.id));
  if (index === -1) return res.status(404).json({ status: 404, error: 'Contacto no encontrado.' });
  contacts.splice(index, 1);
  res.json({ message: 'Contacto eliminado.' });
});

// ── Ruta no encontrada
app.use((req, res) => {
  res.status(404).json({ status: 404, error: 'Ruta no encontrada.' });
});

// ── Error inesperado
app.use((err, req, res, next) => {
  console.error(err.stack);
  res.status(500).json({ status: 500, error: 'Error interno del servidor.' });
});

module.exports = { app, resetContacts };

✏️
Bloque A — Tests de validación con regex de email
Nuevos casos de borde que la validación con includes('@') no cubría

El cambio de includes('@') a regex abre nuevos casos que antes pasaban sin error y ahora deben ser rechazados.

Tests requeridos — POST /api/contacts (validación)
  1. Devuelve 400 cuando el email es "@" (sin usuario ni dominio).
  2. Devuelve 400 cuando el email es "usuario@" (sin dominio).
  3. Devuelve 400 cuando el email es "@dominio.com" (sin usuario).
  4. Devuelve 400 cuando el email es "sin-arroba".
  5. Devuelve 201 cuando el email tiene formato válido "usuario@dominio.com".
  6. El mensaje de error contiene la palabra "email" (verifica con toMatch(/email/i)).
  • Usa request(app).post('/api/contacts').send({ name: 'Test', email: '@' }) para cada caso.
  • Agrupa estos tests dentro de un describe('validación de email', ...) para mantener el archivo organizado.
  • No hace falta testear todos los posibles emails inválidos — solo los que representan caminos distintos en la regex (sin usuario, sin dominio, sin punto, sin arroba).
✏️
Bloque B — Tests de detección de email duplicado
Verifica el comportamiento del 409 y la insensibilidad a mayúsculas

Recuerda que el 409 es semánticamente distinto al 400. Tus tests deben verificar el código exacto, no solo que "haya un error".

Tests requeridos
  1. Crear un contacto con un email ya existente devuelve 409.
  2. El body del 409 tiene el campo error con un mensaje descriptivo.
  3. Crear con email en mayúsculas ("ANA@EXAMPLE.COM") cuando ya existe "ana@example.com" también devuelve 409 (deduplicación case-insensitive).
  4. Después de recibir un 409, el número total de contactos no aumentó (el POST fue un no-op).
🔍
Test 4: verificar que no se creó el contacto

Haz el POST duplicado y luego un GET /api/contacts para contar cuántos hay. Si el resetContacts() en beforeEach funciona bien, siempre deberías partir de 3 contactos.

  • El email "ana@example.com" ya existe en los datos iniciales. Úsalo para el test de duplicado.
  • Para el test 4: const listRes = await request(app).get('/api/contacts'); y luego expect(listRes.body).toHaveLength(3);.
  • El test de case-insensitive es importante porque muchos sistemas guardan los emails normalizados a minúsculas pero los usuarios escriben con mayúsculas.
✏️
Bloque C — Tests de búsqueda y filtros en GET /api/contacts
Query params: ?search= y ?favorite=true, incluyendo combinaciones

Los query params son invisibles para el código si no se testean explícitamente. Un refactor que rompa el filtrado pasaría desapercibido sin estos tests.

Tests requeridos
  1. ?search=ana devuelve solo contactos cuyo nombre o email contiene "ana".
  2. ?search=ANA (mayúsculas) devuelve los mismos resultados (case-insensitive).
  3. ?search=example filtra por email y devuelve todos los que tienen "@example.com".
  4. ?search=xyznoexiste devuelve un array vacío (no un 404).
  5. ?favorite=true devuelve solo contactos con favorite: true.
  6. ?favorite=true devuelve un array vacío si ninguno es favorito (tras resetContacts, solo Luis es favorito).
  7. Sin query params devuelve todos los contactos (comportamiento original sin regresión).
  • Para pasar query params con Supertest: request(app).get('/api/contacts').query({ search: 'ana' }) o directamente en la URL: request(app).get('/api/contacts?search=ana').
  • Recuerda que con los datos iniciales, solo Luis tiene favorite: true. Planea tus asserts en consecuencia.
  • El test 4 (sin resultados) es importante para documentar que la API devuelve [] y no un 404.
  • Para verificar que todos los resultados tienen favorite: true: expect(res.body.every(c => c.favorite)).toBe(true).
✏️
Bloque D — Tests del toggle de favorito (PATCH)
Comportamiento de transición de estado: false→true→false

Un endpoint de toggle tiene una propiedad especial: su resultado depende del estado previo. Necesitas testear ambas transiciones y también el caso de ID inexistente.

Tests requeridos
  1. Llamar PATCH /api/contacts/1/favorite en Ana (que tiene favorite: false) devuelve el contacto con favorite: true.
  2. Llamarlo dos veces sobre el mismo contacto regresa a favorite: false (toggle completo).
  3. Llamarlo en Luis (que inicia con favorite: true) devuelve el contacto con favorite: false.
  4. Devuelve 404 para un ID inexistente.
  5. Después del toggle, un GET /api/contacts/:id refleja el cambio persistido (no es solo en el response del PATCH).
🔍
Test 5 — verificar persistencia del cambio

Este test verifica que el PATCH realmente modificó el array en memoria, no solo devolvió un valor calculado. Haz el PATCH y luego un GET independiente sobre el mismo ID para confirmar que el estado cambió.

  • Para el test del doble toggle: llama PATCH dos veces seguidas y verifica el estado final con un GET.
  • Usa los datos de seed conocidos: Ana (id=1) inicia en false, Luis (id=2) inicia en true.
  • La URL es PATCH /api/contacts/1/favorite — Supertest: request(app).patch('/api/contacts/1/favorite').
✏️
Bloque E — Tests del PUT mejorado
Validación de email al actualizar y deduplicación en edición

El PUT ahora valida el nuevo email y detecta duplicados. Esto añade más rutas de error que no existían antes.

Tests requeridos
  1. Actualizar solo el name devuelve 200 y el contacto con el nombre cambiado.
  2. Intentar actualizar con un email de formato inválido devuelve 400.
  3. Intentar actualizar con el email de otro contacto existente devuelve 409.
  4. Actualizar el email con el mismo email del contacto actual (sin cambio real) devuelve 200 — no debe ser un 409 porque no es un duplicado externo.
  5. Actualizar un ID inexistente devuelve 404.
  • Para el test 3 (duplicado en update): intenta cambiar el email de Ana (id=1) al email de Luis ("luis@example.com").
  • Para el test 4 (mismo email): actualiza Ana con { email: "ana@example.com" } — debería ser 200 porque el código excluye el propio contacto en la búsqueda de duplicados (c.id !== contact.id).
  • Estos tests documentan la diferencia entre "email ya usado por otro" (409) vs "mismo email que ya tengo" (200).
✏️
Bloque F — Tests del middleware de error
Rutas inexistentes y formato uniforme de respuestas de error

El middleware de error es "invisible" en el código feliz. Sin tests explícitos, nadie notaría si se rompe o si una nueva ruta devuelve HTML en lugar de JSON.

Tests requeridos
  1. Hacer GET /api/ruta-que-no-existe devuelve 404 con Content-Type: application/json.
  2. La respuesta del 404 genérico tiene el campo error en el body (no HTML).
  3. Todos los errores de negocio (400, 404, 409) tienen un campo status con el número de código HTTP correspondiente.
🔍
¿Por qué testear el Content-Type?

Si Express no tiene un middleware de error configurado, devuelve HTML por defecto en rutas inexistentes. Un cliente que espera JSON rompería silenciosamente al intentar parsear HTML. Verificar res.headers['content-type'] o usar .expect('Content-Type', /json/) de Supertest detecta este bug.

  • Supertest tiene un shorthand: request(app).get('/no-existe').expect('Content-Type', /json/).
  • Para verificar el campo status en errores: haz un GET /api/contacts/9999 (404) y verifica que res.body.status sea 404.
  • Para el test 3, puedes reutilizar la estructura de tests existentes y agregar expect(res.body).toHaveProperty('status', 404).

Por qué importan estos tests — explicación extendida

Cada bloque de tests que escribiste en este ejercicio cubre un tipo distinto de riesgo. Aquí se explica qué bug detectaría cada uno si el código de producción tuviera un error.

Bloque ¿Qué bug detecta? Sin el test, el bug se manifestaría en...
A · Validación regex Emails malformados que pasan la validación y entran a la BD Producción, cuando un usuario escribe usuario@ y el sistema lo acepta como válido
B · Duplicados (409) Dos contactos con el mismo email, búsquedas ambiguas, confusión de identidad Cuando el soporte intente buscar "el único" cliente con ese email y encuentre dos
C · Búsqueda Filtros que devuelven todos los contactos cuando deberían filtrar, o que no son case-insensitive Un usuario busca "ANA" y no encuentra a "Ana" — parece un bug de UI pero es del API
D · Toggle favorito El toggle que no persiste, o que siempre pone true sin importar el estado previo El usuario marca y desmarca un favorito y parece funcionar, pero al recargar la página sigue marcado
E · PUT mejorado Un update de email que bypasea la validación, o que rechaza el propio email del contacto como "duplicado" Un usuario intenta actualizar su nombre y el sistema le dice que su email "ya existe" (falso positivo)
F · Middleware de error Rutas inexistentes que devuelven HTML; clientes que crashean al hacer JSON.parse de HTML Una app móvil que consume el API y crashea en producción con "SyntaxError: Unexpected token <"
Principio fundamental Los tests no son un lujo — son el contrato del sistema

Cuando alguien lee it('rechaza emails duplicados con 409') en el archivo de tests, aprende tres cosas sin leer el código de producción: (1) el sistema verifica duplicados, (2) el código de respuesta es 409 y no 400, (3) esta es una invariante que debe mantenerse al refactorizar.

Un sistema sin tests es un sistema donde nadie puede refactorizar con confianza. Cada mejora de código se convierte en una apuesta arriesgada.

Tests como documentación viva

A diferencia de los comentarios (que se desactualizan), los tests fallan si el comportamiento cambia. Son documentación que se verifica automáticamente.

Regresiones detectadas antes

Si agregas una nueva ruta y rompes el middleware de 404, el test F lo detecta antes de hacer deploy. Sin tests, lo descubres cuando los usuarios reportan errores.

Semántica HTTP correcta

Testear 409 vs 400 parece obsesivo, pero los clientes del API toman decisiones basadas en el código HTTP. Un 400 y un 409 significan acciones distintas para el cliente.

Aislamiento de estado

resetContacts() en beforeEach garantiza que cada test sea independiente. Sin esto, el orden de ejecución de los tests afecta su resultado — un bug muy difícil de reproducir.

Cobertura de caminos

La deduplicación case-insensitive, el mismo email en PUT, el toggle doble — son caminos de código que solo se ejecutan en situaciones específicas. Sin tests, llegarás a ellos en producción.

Confianza para iterar

Con este conjunto de tests, puedes refactorizar el código de producción (cambiar la regex, la lógica de duplicados) y saber inmediatamente si rompiste algo.

🚀
Siguiente paso sugerido

Una vez que tengas todos los tests pasando, intenta romper intencionalmente el código de producción (comenta la detección de duplicados, elimina la regex, borra el middleware de error) y verifica que npx jest detecta cada rotura. Eso te dará intuición sobre qué tan buena es tu cobertura.
